# FoxSay 现状与 PRD 差距评估报告

> 评估日期：2026-05-10
> 对照文档：`foxsay-prd.md` v0.1

## 评估方法

逐项对照 PRD 第六章 MVP 功能清单、第三章用户旅程、第五章技术架构、第八章设计语言，对当前代码实现进行诚实评估。评分标准：

| 完成度 | 含义 |
|--------|------|
| **100%** | 完全达到 PRD 描述，包括体验细节 |
| **70-90%** | 核心功能到位，体验或边界有缺口 |
| **30-60%** | 有基本实现，但关键能力缺失 |
| **0-20%** | 仅有骨架或完全缺失 |

## 四、知识图谱专题分析

### 4.1 当前构建方式

唯一写入路径在 `backend/app/services/skeleton.py:88-96`：

```python
for concept in core_concepts:
    concept_id = concept.replace(" ", "_").lower()
    kg.add_concept(concept_id, label=concept)
```

问题清单：

1. **不是知识抽取，是骨架图副产品**：先让 LLM 生成骨架图 JSON（chapters + core_concepts + prerequisite_chain），再把 core_concepts 塞进图。图谱不是从材料中独立构建的。
2. **截断严重**：`materials_text[:8000]`，约 2-3 页 PDF 的文本量参与图谱构建，几百页课程材料的剩余部分对图谱不可见。
3. **概念节点是空壳**：只有 `label`，没有定义、来源 chunk、属性。问"什么是 X"图谱回答不了。
4. **关系只有 prerequisite 一种**：没有 contains、causes、relates_to 等关系，信息密度极低。
5. **没有增量**：新材料上传后图谱不更新。只在课程首次变 ready 时构建一次。
6. **fallback 不建图**：`_fallback_generate` 只按文本分块生成章节，不向图谱添加任何数据。

### 4.2 当前利用方式

**答案：几乎没有被利用。**

| 消费方 | 是否读图谱 | 说明 |
|--------|----------|------|
| `crag.py` ask() | ❌ | 纯向量检索 → LLM 生成 |
| `retrieval.py` retrieve() | ❌ | 纯向量搜索 |
| `review.py` generate_review_plan() | ❌ | 只读 skeleton 快照，不读图谱 |
| `skeleton.py` generate_skeleton() | ✅ 写入 | 唯一写入点，但只是副产品 |

`KnowledgeGraph` 类没有 `query()`、`search()`、`get_neighbors()`、`get_path()` 等检索方法。即使图里有数据，也没有代码能查询它。

### 4.3 根本结论

当前知识图谱是一个**只有写入、没有读取的空转系统**。如果拿掉 `KnowledgeGraph`，直接把 LLM 返回的 JSON 塞进 `CourseSkeleton`，功能完全一样。图谱的实际价值为零。

## 六、综合完成度

### 按功能权重估算

| 功能域 | PRD 权重 | 完成度 | 加权得分 |
|-------|---------|--------|---------|
| 课程创建 + 课程表导入 | 10% | 85% | 8.5% |
| 材料上传与处理 pipeline | 25% | 40% | 10% |
| 课程骨架图 | 15% | 30% | 4.5% |
| 课程内问答（CRAG） | 25% | 70% | 17.5% |
| 超级备考模式 | 15% | 35% | 5.25% |
| 产品体验（人设/引导/主动推送） | 10% | 5% | 0.5% |

**综合完成度：约 46%**

如果只看"后端 API 能跑 + 前端页面能看"的表层完成度，数字会更高。但 PRD 的核心承诺——"课程边界内的全知"、"第一个惊喜"、"贱贱的小狐狸"、"知识网络驱动"——这些都没有落地。当前实质上是一个**支持课程级知识库的 RAG 问答系统**，离 PRD 描述的产品还有本质差距。

### 最关键的三个差距

1. **知识图谱**：构建和利用都为零。这是 PRD 架构的核心差异化层，也是 LightRAG 双层检索的前提。
2. **产品体验**：没有人设、没有引导、没有主动推送。PRD 描述的是有性格的产品，当前是功能骨架。
3. **材料处理深度**：pdfplumber 纯文本提取无法支撑理工科材料，Docling/Marker 未接入。
